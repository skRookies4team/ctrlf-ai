## 1. 문서 개요

### 1.1 목적

본 문서는 CTRL+F 프로젝트의 비기능 요구사항(Non-Functional Requirements, NFR)을 정의합니다. **업계 표준 및 벤치마크를 기반**으로 성능 기준을 설정하고, **비즈니스 요구사항에 맞춰** 시스템의 품질 속성을 명확히 합니다.

### 1.2 범위

| 레포지토리 | 역할 |
|-----------|------|
| ctrlf-front | React 기반 웹 프론트엔드 |
| ctrlf-back | Spring Boot 기반 메인 백엔드 (마이크로서비스) |
| ctrlf-ai | FastAPI 기반 AI 게이트웨이 |
| ctrlf-ragflow | RAG 파이프라인 (문서 처리/벡터화) |

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| NFR | Non-Functional Requirements (비기능 요구사항) |
| SLA | Service Level Agreement (서비스 수준 협약) |
| RTO | Recovery Time Objective (복구 목표 시간) |
| RPO | Recovery Point Objective (복구 목표 시점) |
| TPS | Transactions Per Second (초당 처리량) |
| P95/P99 | 95/99 백분위수 응답 시간 |
| TTFB | Time To First Byte (첫 바이트 응답 시간) |
| TTFT | Time To First Token (첫 토큰 응답 시간) |

## 2. 비즈니스 요구사항 정의

### 2.1 서비스 개요

CTRL+F는 **기업용 AI 기반 교육/정보 검색 플랫폼**으로, RAG(Retrieval-Augmented Generation) 기술을 활용하여 사내 규정, 교육 콘텐츠, 정보보안 문서 등을 검색하고 질의응답을 제공합니다.

### 2.2 타겟 사용자 및 규모

| 항목 | 설정값 | 근거 |
|------|--------|------|
| **대상 기업 규모** | 중견기업 (500~1,000명) | SK 쉴더스 고객사 타겟 |
| **등록 사용자 수** | 1,000명 | 전 임직원 |
| **일일 활성 사용자 (DAU)** | 200~300명 | 등록 사용자의 20~30% |
| **동시 접속 사용자** | 50~100명 | DAU의 20~30% (피크 시간대) |
| **피크 시간대** | 09:00~11:00, 14:00~16:00 | 업무 시작/오후 집중 시간 |

### 2.3 핵심 기능 우선순위

| 우선순위 | 기능 | 중요도 | 사용 빈도 |
|---------|------|--------|----------|
| 1 | AI 챗봇 질의응답 | 매우 높음 | 일 500~1,000회 |
| 2 | 사내 규정 검색 | 높음 | 일 200~500회 |
| 3 | 교육 콘텐츠 조회 | 높음 | 일 100~300회 |
| 4 | 퀴즈/FAQ 생성 | 중간 | 주 10~50회 (관리자) |
| 5 | 교육 영상 생성 | 낮음 | 월 5~20회 (관리자) |

### 2.4 사용 시나리오

**시나리오 1: 사내 규정 문의 (가장 빈번)**
```
임직원 A: "재택근무 신청 절차가 어떻게 되나요?"
→ AI 검색 (1~3초) → LLM 응답 생성 (5~10초) → 답변 표시
→ 총 예상 시간: 10초 이내
```

**시나리오 2: 교육 이수 확인 (빈번)**
```
임직원 B: 정보보안 교육 페이지 접속 → 교육 영상 시청 → 퀴즈 응시
→ 페이지 로딩: 2초 이내, 영상 버퍼링: 3초 이내
```

**시나리오 3: FAQ 생성 (관리자, 간헐적)**
```
관리자 C: FAQ 생성 요청 → 배치 처리 → 검토 → 승인
→ 비동기 처리 허용, 30초~1분 소요 가능
```

## 3. 성능 요구사항 (Performance)

### 3.1 업계 표준 기준

#### 3.1.1 웹 응답 시간 기준 (Jakob Nielsen, 1993 / Google RAIL, 2015)

| 응답 시간 | 사용자 인식 | 적용 대상 |
|----------|-----------|----------|
| **< 100ms** | 즉각적 반응, 지연 인지 불가 | 버튼 클릭, UI 반응 |
| **100ms ~ 1초** | 지연 인지되나 흐름 유지 | API 호출, 검색 |
| **1초 ~ 3초** | 지연 인지, 집중력 유지 가능 | 페이지 로딩 |
| **3초 ~ 10초** | 집중력 저하, 이탈 위험 | 복잡한 처리 |
| **> 10초** | 사용자 이탈 | 허용 불가 |

> **출처**: [MDN Web Performance Guide](https://developer.mozilla.org/en-US/docs/Web/Performance/Guides/How_long_is_too_long)
> "사용자 상호작용에 대해 100ms 이내, 가급적 50ms 이내에 피드백을 제공해야 즉각적으로 느껴집니다."

#### 3.1.2 LLM 응답 시간 벤치마크 (2025년 기준)

| 모델 | 첫 토큰 지연 (TTFT) | 전체 응답 시간 | 비고 |
|------|-------------------|--------------|------|
| OpenAI o3-mini | 1~2초 | 9.5초 | 80% 성공률 |
| Claude 3.5 Sonnet | 1~3초 | 10~15초 | 고품질 응답 |
| Gemini 2.5 Flash | < 1초 | 5~8초 | 최고 속도 |
| 로컬 LLM (7B) | 2~5초 | 10~20초 | 하드웨어 의존 |

> **출처**: [AIMultiple LLM Latency Benchmark](https://research.aimultiple.com/llm-latency-benchmark/)
> "2025년 최신 모델(Gemini 2.5 Flash, o3-mini)은 2023년 대비 5~10배 빠른 응답 속도를 보입니다."

#### 3.1.3 Core Web Vitals (Google, 2024)

| 지표 | Good | Needs Improvement | Poor |
|------|------|-------------------|------|
| **LCP** (Largest Contentful Paint) | ≤ 2.5초 | 2.5~4초 | > 4초 |
| **FID** (First Input Delay) | ≤ 100ms | 100~300ms | > 300ms |
| **CLS** (Cumulative Layout Shift) | ≤ 0.1 | 0.1~0.25 | > 0.25 |
| **INP** (Interaction to Next Paint) | ≤ 200ms | 200~500ms | > 500ms |

> **출처**: [Google Web Vitals](https://web.dev/vitals/)

### 3.3 처리량 (Throughput)

| 서비스 | 목표 TPS | 피크 TPS | 동시 사용자 | 근거 |
|--------|---------|---------|------------|------|
| ctrlf-front | 100 req/s | 200 req/s | 100명 | 사용자당 1~2 req/s |
| ctrlf-back | 50 req/s | 100 req/s | 100명 | AI로 집중되는 구조 |
| ctrlf-ai | 10 req/s | 20 req/s | 20명 | LLM 병목 (응답 10초 가정) |
| Milvus | 50 req/s | 100 req/s | - | 벡터 검색 전용 |

**계산 근거**:
```
1. 동시 사용자 100명 가정 (DAU 300명의 피크 시간대 33%)
2. 사용자당 평균 요청: 1~2 req/s (페이지 조회 + API 호출)
3. ctrlf-ai TPS = 동시 사용자 20명 × (1 요청 / 10초 응답) = 2 req/s (여유 있게 10 req/s)
4. AI 채팅은 응답 완료까지 10초 소요 → 동시 처리 20건 = 200초 분량
```

### 3.4 재시도 정책 (Retry Policy)

| 서비스 | 최대 재시도 | 기본 지연 | 최대 지연 | 근거 |
|--------|-----------|----------|----------|------|
| RAGFlow | 1회 | 0.2초 | 1.0초 | 검색 실패 시 빠른 재시도 |
| LLM | 1회 | 0.5초 | 2.0초 | 비용/지연 균형 |
| Backend API | 1회 | 0.2초 | 1.0초 | 일시적 장애 복구 |
| S3 업로드 | 3회 | 1.0초 | 8.0초 | 네트워크 불안정 대응 |

**재시도 알고리즘**: 지수 백오프 (Exponential Backoff)
```python
delay = base_delay × (2 ^ attempt)  # 0.2, 0.4, 0.8, ...
delay = min(delay, max_delay)       # 최대 지연 제한
```

**근거**:
```
1. 재시도 1회: 과도한 재시도는 장애 전파 위험 (Thundering Herd)
2. 지수 백오프: AWS, Google Cloud 권장 패턴
3. S3 3회 재시도: AWS SDK 기본값
```

## 4. 가용성 요구사항 (Availability)

### 4.1 업계 표준 SLA

| SLA 수준 | 연간 다운타임 | 월간 다운타임 | 일반 적용 대상 |
|---------|-------------|-------------|--------------|
| 99% (Two Nines) | 3.65일 | 7.31시간 | 내부 도구, 개발 환경 |
| 99.5% | 1.83일 | 3.65시간 | 비핵심 SaaS |
| **99.9% (Three Nines)** | **8.77시간** | **43.83분** | **일반 SaaS (업계 표준)** |
| 99.95% | 4.38시간 | 21.92분 | E-commerce, 고객 대면 |
| 99.99% (Four Nines) | 52.6분 | 4.38분 | 금융, 의료, 엔터프라이즈 |

> **출처**: [Uptime.is SLA Calculator](https://uptime.is/)
> "99.9%는 업계에서 가장 일반적인 SLA 수준으로, 거의 하루에 해당하는 장애를 허용합니다."

### 4.2 CTRL+F 가용성 목표

| 환경 | 목표 SLA | 월간 허용 다운타임 | 근거 |
|------|---------|------------------|------|
| 개발 (Development) | 95% | 36시간 | 개발 중 빈번한 재배포 |
| 스테이징 (Staging) | 99% | 7.3시간 | 테스트 환경 |
| **운영 (Production)** | **99.9%** | **43.8분** | **SaaS 업계 표준** |

**99.9% 선정 근거**:
```
1. 업계 표준: 대부분의 SaaS 벤더(Salesforce, Slack 등)가 99.9% 제공
2. 비용 효율: 99.99% 달성에는 막대한 인프라 투자 필요
3. 비즈니스 영향: 기업 내부 도구로 월 43분 다운타임 허용 가능
4. Oracle SaaS 기준: 99.9% 이상 (출처: IT Convergence)
```

### 4.3 헬스체크 설정

| 파라미터 | 설정값 | 근거 |
|---------|--------|------|
| 체크 간격 (interval) | 30초 | 빠른 장애 감지 vs 부하 균형 |
| 타임아웃 (timeout) | 10초 | 네트워크 지연 허용 |
| 시작 유예 (start-period) | 5초 | 컨테이너 초기화 시간 |
| 실패 허용 (retries) | 3회 | 일시적 장애 허용 |

**장애 감지 시간 계산**:
```
최대 장애 감지 시간 = interval × retries = 30초 × 3 = 90초
→ 장애 발생 후 최대 90초 이내 감지
```

### 4.4 장애 복구 목표

| 지표 | 목표 | 근거 |
|------|------|------|
| **RTO** (Recovery Time Objective) | 30분 | Kubernetes 자동 복구 + 수동 대응 |
| **RPO** (Recovery Point Objective) | 15분 | PostgreSQL WAL 아카이빙 주기 |
| **MTTR** (Mean Time To Recovery) | 15분 | 자동 복구 기준 |
| **MTBF** (Mean Time Between Failures) | 720시간 (30일) | 99.9% SLA 역산 |

**MTBF 계산**:
```
월간 허용 다운타임 = 43.8분
월간 운영 시간 = 30일 × 24시간 = 720시간
MTBF = 720시간 / (43.8분 / 60분) ≈ 986시간 (약 41일)
→ 보수적으로 30일(720시간) 설정
```
